# 10 Cursor Rules That Save Me 5 Hours Every Week

*By [Your Name] - Vibe Academy*

As someone who's written over 100,000 lines of code with AI assistance, I've discovered that the right Cursor rules can transform your productivity. Here are my top 10 rules that consistently save me 5+ hours per week.

## 1. The Component Generator Rule
```
When I ask you to create a React component, always:
- Use TypeScript
- Include proper interfaces/types
- Add JSDoc comments
- Export at the bottom
- Include basic error handling
```

**Time saved**: 20 minutes per component

## 2. The Test Writer Rule
```
When writing tests:
- Use descriptive test names
- Include edge cases
- Add setup and teardown
- Test both success and failure paths
- Include accessibility tests when relevant
```

**Time saved**: 30 minutes per feature

## 3. The API Endpoint Rule
```
For API routes, always:
- Include request/response types
- Add input validation
- Include error handling
- Return consistent response format
- Add rate limiting comments
```

**Time saved**: 15 minutes per endpoint

## 4. The Database Query Rule
```
When writing database queries:
- Use prepared statements
- Include error handling
- Add query performance comments
- Return typed results
- Include transaction handling when needed
```

**Time saved**: 25 minutes per feature

## 5. The Refactoring Assistant Rule
```
When refactoring code:
- Preserve all functionality
- Improve readability first
- Add missing types
- Extract magic numbers to constants
- Split large functions
```

**Time saved**: 45 minutes per refactor

## 6. The Documentation Rule
```
For documentation:
- Write clear JSDoc comments
- Include usage examples
- Document all parameters
- Add return type descriptions
- Include common gotchas
```

**Time saved**: 20 minutes per module

## 7. The Debug Helper Rule
```
When debugging:
- Add descriptive console logs
- Include variable states
- Show function entry/exit
- Add timing measurements
- Suggest likely causes
```

**Time saved**: 30 minutes per bug

## 8. The Code Review Rule
```
When reviewing code:
- Check for security issues first
- Verify error handling
- Suggest performance improvements
- Ensure consistent style
- Recommend better naming
```

**Time saved**: 15 minutes per review

## 9. The Git Commit Rule
```
For git commits:
- Use conventional commit format
- Keep subject line under 50 chars
- Include why, not just what
- Reference issue numbers
- List breaking changes
```

**Time saved**: 5 minutes per commit

## 10. The Project Setup Rule
```
When starting new projects:
- Include TypeScript config
- Add ESLint and Prettier
- Set up basic folder structure
- Include README template
- Add basic CI/CD config
```

**Time saved**: 60 minutes per project
